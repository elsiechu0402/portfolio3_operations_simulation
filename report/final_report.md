# 1. Introduction

Warehouse order fulfillment usually includes several connected processes, such as picking, packing, and shipping. If one process has lower capacity than the other processes, this can cause orders to wait in line and form a bottleneck. This phenomenon can lead to many problems, such as increased waiting time, delayed order completion, and limited throughput.

Adding more workers or shipping stations does not always amend the whole operations system. Since if additional resource is not complement the bottleneck, the main problem may still remain there. Therefore, it is necessary to identify where the actual bottleneck is before making resource allocation decisions.

In this project, I built a discrete-event simulation model of a warehouse order fulfillment system with SimPy package in Python . The model includes necessary process, such as order arrival, picking, packing, and shipping. From comparing 5 resource configurations with staffing and capacity senario analysis , I assessed system-level performance metrics by waiting time, completion time, throughput, and resource utilization. I then set 4 demand intensity levels in the sensitivity analysis. 
The purpose of this project is to show how simulation can be used to identify operational bottlenecks and support warehouse staffing and capacity planning.

# 2. Research Question

Main research question:

How do different resource configurations affect waiting time, completion time, throughput, and resource utilization in a simulated warehouse system?

Related questions:

1. Which process stage is the main bottleneck under the baseline configuration?
2. Among the staffing and capacity scenarios tested, which configuration produces the largest improvement in warehouse performance?
3. How does the baseline system perform under different demand levels?

# 3. System Description

## 3.1 Warehouse Process

The simulated warehouse includes three main order fulfillment processes: picking, packing, and shipping. After an order arrives, it first step is waitting for an available picker. When picking process is completed, the order moves to the second packing stage and waits for an available packer. It then enters the third shipping stage and leaves the system after shipping process is completed.

Since each stage has limited resources, orders need to wait when allocated worker(s) or station(s) are unavailable. For each order, the simulation records the waiting time and processing time at the picking, packing, and shipping stages, as well as the total completion time.

## 3.2 Baseline System

The baseline system represents one 480-minute warehouse shift. Orders arrive with an average interarrival time of 5 minutes. The warehouse has two pickers, one packer, and one shipping station. The average picking time is 8 minutes, the average packing time is 6 minutes, and the average shipping time is 4 minutes. Order arrival times and processing times are randomly generated, so each simulation run can produce different results.

**Table 1. Baseline warehouse simulation parameters.**

| Number of pickers | 2 |

## 3.3 Model Assumptions

This system is a simplified form of warehouse operations. All orders go through the same three stages: picking, packing, and shipping. Orders are processed in the order as they arrive, and workers and stations remain available state during wholel simulation process. However, this system does not include worker breaks, shift changes, equipment failures, inventory shortages, order cancellations, or rework, etc. It also does not consider differences in order size, product type, or priority. The data are generated by the simulation and are not taken from a real warehouse management system.

# 4. Methods

## 4.1 Simulation and Replications

The warehouse simulation was built in Python using SimPy, a process-based discrete-event simulation framework (Team SimPy 2026). Order arrival times and the processing times for picking, packing, and shipping were randomly generated.

Because simulation output is affected by random variation, multiple replications were used to estimate average system performance (Law 2015). Each setting was simulated 30 times using different random seeds. The average results across the 30 replications were then used for comparison.

## 4.2 Scenario and Bottleneck Analysis

Five resource configurations were compared: Baseline, Add One Picker, Add One Packer, Add One Shipping Station, and Balanced Capacity. Only the numbers of pickers, packers, and shipping stations were changed in every senarios. The simulation time, order arrival setting, and processing-time parameters were kept the same across the five scenarios.

**Table 2. Staffing and capacity scenarios.**

| Scenario | Pickers | Packers | Shipping stations |
|---|---:|---:|---:|
| Baseline | 2 | 1 | 1 |
| Add One Picker | 3 | 1 | 1 |
| Add One Packer | 2 | 2 | 1 |
| Add One Shipping Station | 2 | 1 | 2 |
| Balanced Capacity | 3 | 2 | 1 |

Table 2 shows how the numbers of pickers, packers, and shipping stations were changed across the five tested configurations.

I then compared resource utilization and waiting time before each stage to identify the bottleneck. A resource with high utilization and a long queue waiting time was considered a potential system constraint. The changes in throughput, total waiting time, and completion time were also compared with the baseline results.

## 4.3 Demand Sensitivity Analysis

A separate sensitivity analysis was conducted using the baseline resource configuration of two pickers, one packer, and one shipping station. The average time between order arrivals was changed to 6, 5, 4, and 3.5 minutes. A shorter interarrival time represents a higher demand level.

Each demand with different interarrival time was also simulated 30 times. The average results were compared to examine how changes in demand affected throughput, waiting time, completion time, and resource utilization.

# 6. Discussion

The simulation results show that warehouse performance depends on where the additional capacity is added. In the baseline configuration, the packer had the highest resource utilization(0.947), and orders had the longest waiting time before packing(42.093). These two indicators show that the packing stage was the main operational bottleneck.

Adding one picker did not produce a clear improvement because picking was not the main constraint in the baseline system. The additional picker reduced picker utilization, but orders still had to wait for the same single packer. As a result, the packing queue remained long, and the improvements in throughput, total waiting time, and completion time were limited.

Adding one shipping station also did not produce improvement in simulation system. Shipping capacity increased, but packing capacity remained previous level. The additional shipping station therefore could not solve the main problem of congestion. This shows that adding a resource to a non-bottleneck stage may increase the processing capacity of that stage without improving overall system performance.

In contrast, from the lowest average total waiting time(8.643) , the results show that adding one packer significantly increased system effiency at the bottleneck. By reducing time of packing queue so that system can allow more orders into fulfillment process. Among three single-resource addition scenarios, Add One Packer produced the largest improvement in throughput, total waiting time(8.643), and completion time(26.716).

Balanced Capacity achieved the best overall performance among the five tested configurations. After adding one picker and one packer, the workload achieve a more rational allocation across the three stages. However, shipping became the highest-utilization resource and the longest-wait stage. This shows that the main constraint can alter after the initial bottleneck is reduced. Therefore, capacity decisions should consider the whole process rather than evaluate each stage separately.

The demand sensitivity analysis also shows the capacity limit of the baseline system. As the average interarrival time became shorter, orders entered the system more frequently. Throughput increased slightly from 6.0 minutes of interarrival time to 3.5 minutes of interarrival time, while total waiting time and completion time increased substantially from lower demand level to higher demand level. This indicates that the baseline resources could not deal with additional demand at the same rate with more entering orders.

Overall, the results show that simulation can support warehouse staffing and capacity planning by identifying the main bottleneck before resources are added. The analysis also shows that the main bottleneck can shift after additional resources are added, while the baseline configuration performs less effectively as demand increases.

# 7. Limitations

This project has several limitations. Firstly, the model used simulated data rather than data from a real warehouse management system. The order arrival and processing-time parameters were assumed for this somulation study rather than estimated from actual warehouse records. Therefore, the results should be interpreted as a comparison of the tested scenarios rather than a recommendation for a specific warehouse.

Secondly, all orders follow the same picking, packing, and shipping process. This warehouse simulation model does not distinguish orders by size, product type, priority, or handling requirements. In a real warehouse, these differences may affect processing time, resource utilization, and queue length.

Also, the model assumes that all workers and stations remain available during the whole simulation period, it did not include worker breaks, shift changes, equipment failures, inventory shortages, order cancellations, and rework. Warehouse layout and travel distance were also excluded, even they may have an significant effect on picking time.

Moreover, the scenario analysis only includes five resource configurations. Other combination modes of pickers, packers, and shipping stations may produce different results. The demand sensitivity analysis only evaluates the baseline configuration under different average interarrival times, it does not compare all five resource configurations for different demand values.

Lastly, this project evaluates operational performance, including throughput, waiting time in every process, completion time in every process, utilization of every stage, but does not including labor, equipment, or workstation costs. Balanced Capacity achieved the best operational performance among the tested configurations, but the results do not show whether it is the most cost-effective option. In further analysis it would be meaningful to see combining simulation results with cost data to compare the operational benefits and financial costs of different resource configurations.

# 8. Conclusion

This project developed a discrete-event simulation model to examine staffing, bottlenecks, and demand intensity in a warehouse order fulfillment system. Five resource configurations were compared by throughput, waiting time, completion time, and resource utilization.

The results show that packing was the main operational bottleneck in the baseline configuration. In the baseline configuration, the packer had the highest resource utilization (0.947), and orders had the longest average waiting time before the packing stage (42.093). Adding one picker or one shipping station did not improve efficiency significantly because these changes did not increase processing capacity of the bottleneck stage - Packing.

Among these three single-resource addition scenarios, Add One Packer achieved the best performance improvement, and Balanced Capacity achieved the best overall operational performance among the total five tested resource configurations. However, after additional resources were added, the main constraint shifted to another stage. This shows that warehouse processes should be evaluated as a connected whole system rather than separate parts.

The demand sensitivity analysis also illustrates that the baseline configuration becomes less effective as orders arrive more frequently. At higher demand levels, waiting time and completion time increased substantially, while throughput increased slightly. This indicates that when order demand increases while the baseline resource configuration remains unchanged, additional demand mainly causes longer waiting times and completion times because the system has limited additional processing capacity. And throughput increases only slightly. Therefore, the main bottleneck should be identified, and staffing and capacity should be adjusted before demand increases.

Overall, this project indicates that discrete-event simulation can support warehouse staffing and capacity planning by identifying the main bottleneck and comparing alternative resource configurations before changes are implemented. Further analysis could use real warehouse data and include operating costs to compare both operational performance and cost-effectiveness.
